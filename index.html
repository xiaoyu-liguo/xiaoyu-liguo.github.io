一、 注入灵魂：将布局绑定到 Activity

布局图纸画好了，接下来就是让它进入 App 的生命周期里。在 Android 中，持有页面布局的组件是 Activity。

💡 核心认知： Activity 只是 View 的宿主（持有者），并不是绘画者。它不仅管 UI，还要管生命周期、系统事件等。所以，很多新手把“Activity”直接等同于“一个页面布局”，其实是非常不准确的。

但是**Activity 类的源码里，压根就没有一个属性是用来存 View 的！**（没有类似 View mRootView 或者 DecorView mDecor 这样的全局变量）。至于他们的关系，可见视图的真实引用链

要把刚才的 XML 注入给 Activity，现代 Android 开发通常有两种流派：

方法 1：传统手艺（findViewById）

  • 方法 1（传 ID）：只给了 Activity 一个极其轻量的**“图纸编号”**（R.layout.first_layout 本质上是个 int 数字）。此时 XML 还没有被加载到内存中。是 Activity 内部暗中调用了 LayoutInflater，极其辛苦地拿着编号去硬盘里找 XML，并把它“充气”成内存里的 View。

方法 2：现代标配（ViewBinding 视图绑定）

  • 方法 2（传 View）：在调用 setContentView 之前，通过 FirstLayoutBinding.inflate(layoutInflater) 提前召唤了 LayoutInflater，把图纸变成了真正的、在内存里建好的**“现房”**（binding.root 本质上是个 View）。然后你直接把现房丢给了 Activity。

1.1 getWindow()是什么？

Window 的真面目：它是“制定规则的宪法”

你贴出的这几行源码（FEATURE_NO_TITLE、FEATURE_OPTIONS_PANEL 等），正好暴露了 Window 的真实身份：它是一个“顶级视图策略（Policy）”的定义者。

它就像一份合同或者图纸，规定了一个“窗口”应该具备哪些能力：

  • 有没有标题栏？（FEATURE_NO_TITLE）
  • 左边有没有图标？（FEATURE_LEFT_ICON）
  • 背景是不是透明的？
  • 全屏还是非全屏？

这也是为什么你在早期的 Android 开发中，如果想要隐藏标题栏，必须在 setContentView 之前调用一句话： requestWindowFeature(Window.FEATURE_NO_TITLE); 这就是你在向 Window 这个大总管申请：“我要修改这份合同上的第 1 条规则！”

“唯一的太子”：PhoneWindow

既然 Window 只是个签合同的“虚职”（抽象类），那总得有人来干脏活累活吧？

Google 在设计 Android 时，可能幻想着未来会有 TvWindow（电视窗口）、WatchWindow（手表窗口）、CarWindow（汽车窗口）来继承这个抽象类。

但现实极其骨感： 整个 Android 源码中，Window 这个庞大的抽象家族里，只有一个唯一且真实的子类（太子），它的名字叫 PhoneWindow。

当你调用 Activity.getWindow() 时，底层其实是通过多态，返回了一个实实在在的 PhoneWindow 对象！所有的界面绘制、按键事件分发、XML 解析，全都是这个 PhoneWindow 在默默执行。

2.1 为什么必须是找这个空盒子

  • 防重叠（占位）： 如果系统直接把你的 XML 布局塞进 DecorView，那你的页面就会直接顶到屏幕最上面，和头顶的 ActionBar（标题栏）重叠在一起！为了防止这种尴尬，系统先用一个垂直的 LinearLayout 把屏幕上下分好区：上面放标题栏，下面留一个叫 content 的 FrameLayout 给你。

  • 主题控制： 早年 Android 开发时，如果想隐藏标题栏，必须在 setContentView 之前调用 requestWindowFeature(Window.FEATURE_NO_TITLE)

    • 为什么必须在它之前？

    • 因为当你写了 NO_TITLE 时，系统在建这栋房子时，就不会往里面塞 ActionBar 了，而是直接给一个占据全屏的 content 空盒子。如果房子已经建好了你再去改，就晚了！

很多老司机在写全局弹窗（比如一个覆盖全屏的加载动画、或者一个全局的悬浮按钮）时，不知道该把这个 View 添加到哪里。因为如果加到自己写的 LinearLayout 里，它可能会被限制在某个局部区域。

终极解决方案： 直接在代码里写 Activity.findViewById(android.R.id.content)！ 因为这玩意儿是系统专门留给你塞内容的最大容器。拿到这个容器，你就可以用 addView() 强行把你的悬浮按钮塞进去，它绝对能覆盖在你自己写的普通 XML 布局之上！

2.2 LayoutInflater 的“造物魔法” (Inflate)

为什么叫 Inflate（充气）？因为 XML 文件在硬盘里只是一堆干瘪的文本字符串（比如 <Button>）。LayoutInflater 的工作，就是给这些文本“充气”，让它们变成内存里丰满的、活生生的 Java/Kotlin 对象。

它干活的流程极其硬核：

  1. 解析 XML (XmlPullParser)：它像一个无情的阅读机器，一行一行地读取你的 XML 文件，遇到一个 <Button> 标签，它就记录下来。
  2. 反射造物 (Reflection)：它是怎么把字符串 <Button> 变成内存里的 android.widget.Button 对象的？答案是反射！它在底层拼接出完整的类名，然后通过 Class.forName().newInstance()，硬生生地把对象 new 出来。
  3. 拼装树结构：它按照 XML 里的嵌套关系，把子 View（比如 Button）通过 addView() 塞进父容器（比如 LinearLayout），最后再把这整棵树，塞进那个 contentParent 空盒子里。

💡 性能优化的真相 为什么“XML 布局层级为什么不能太深”？为什么现在要主推 Jetpack Compose 和 ViewBinding？ 因为 IO 读取 XML + 递归解析 + 大量反射 new 对象，这套组合拳极其耗时！层级越深，反射越慢，页面启动就越卡！

3 这些View在DecorView怎么存的？

3.1 源码铁证：它就藏在 ViewGroup 里

因为 DecorView 继承自 FrameLayout，而 FrameLayout 又继承自 ViewGroup。所以这个存子节点的“盒子”，其实是写在所有容器的老祖宗 ViewGroup.java 里面的。

没错，不管是 DecorView 里的状态栏背景和 mSubDecor，还是你自己写的 LinearLayout 里的几个 Button，它们在内存里的本质，就是挨个排在这个 View[] 数组里面的对象引用。

3.2 🤔 深度架构思考：为什么用数组，坚决不用链表？

“既然界面的 View 是动态添加（addView）和删除（removeView）的，链表的插入和删除是 O(1)，数组是 O(n)，那为什么 Google 偏偏要用数组？

  1. 渲染性能的生死线：CPU 缓存友好性（Cache Locality）

手机屏幕一秒钟要刷新 60 次（甚至 120 次）。这意味着系统每隔 16 毫秒，就要把整棵视图树极其疯狂地遍历一遍（执行 measure、layout、draw）。

  • 链表： 节点在内存里是随机、分散存放的。CPU 去读链表时，缓存命中率极低，找下一个节点需要不停地去主内存里寻址，极其拖慢渲染速度。
  • 数组： 在内存中是连续分配的！CPU 在读取第 0 个 View 时，会顺便把后面的第 1、2、3 个 View 全加载到 L1/L2 高速缓存里。遍历数组的速度，在底层硬件层面简直是碾压链表的。
  1. 极致的随机访问速度：O(1)

你在代码里经常会用到 ViewGroup.getChildAt(int index) 这个方法吧？

因为底层是 mChildren 数组，系统只需要执行 mChildren[index]，瞬间就能拿到对应的 View。如果是链表，还得从头开始极其痛苦地遍历过去。

  1. 动态扩容机制（类似 ArrayList）

既然是数组，容量不够了怎么办？

ViewGroup 内部实现了一个极其高效的动态扩容机制。当 addView() 发现 mChildren 满了时，它会极其果断地 new 一个更大的数组（通常是原来容量的 1.5 倍或特定步长），然后用底层极其高效的 native 方法 System.arraycopy()，把老数组的数据瞬间“拍”到新数组里。

💡 架构师的思考：为什么要这么设计？

Google 为什么要搞这么麻烦？为什么不直接在 Activity 里写一个 View mView？

这其实是极其经典的面向对象设计原则：单一职责原则（SRP）与解耦。

  1. Activity 的核心职责是“管事”：它负责和底层的操作系统（AMS）打交道，接收生命周期（启动、暂停、销毁），处理跨组件通信（Intent）。它应该是一个逻辑控制器。

源码解密：从 ActivityThread 到 SurfaceFlinger，Android 到底是怎么画出界面的？

很多开发者以为，在 Activity 的 onCreate 里调用完 setContentView，屏幕上就应该有画面了。但如果此时立刻去获取一个 View 的宽高（view.getWidth()），你会发现结果永远是 0！ 因为在 onCreate 阶段，系统只是在内存里建好了一棵枯燥的 View 树，根本还没有开始画！”

答案就藏在 Android 主线程的大总管 —— ActivityThread.java 里。系统在准备回调 Activity 的 onResume() 时，才真正吹响了渲染的冲锋号。

揭秘 a.getWindowManager() 的真面目

在代码里，a 就是我们的 Activity。当我们调用 a.getWindowManager() 时，返回的类型虽然写着 ViewManager（或者是它的子接口 WindowManager），它的真实实现类 —— WindowManagerImpl

  1. 它是 Activity 的“私人大堂经理”

每当系统启动一个 Activity（在底层的 Activity.attach() 方法里），系统不仅会给它发一个 PhoneWindow，还会专门为这个 Activity new 一个专属的 WindowManagerImpl。

这个大堂经理手里死死攥着两样东西：

  • 你当前的 Context（也就是这个 Activity）
  • 你当前的屏幕设备（因为有的手机可能外接了显示器，它得知道画在哪块屏幕上）
  1. 为什么要有这个大堂经理？

如果所有 Activity 都直接去找全局系统服务，一旦你把 Activity 销毁了，系统如果忘记清理，就会导致极其严重的内存泄漏。 所以，Google 设计了这个“大堂经理”（WindowManagerImpl）。你 Activity 要加弹窗、加视图，都找你的私人经理。等你 Activity 销毁时，这个经理会负责把你在这个页面上加的所有 View 全部清理掉！

3. 大堂经理背后的“终极 Boss”

私人经理 WindowManagerImpl 其实也是个“甩手掌柜”。当你调用他的 addView 时，他会在内部直接呼叫一个叫 WindowManagerGlobal 的家伙。 WindowManagerGlobal 是一个全局单例（整个 App 进程只有他一个）。所有的跨进程通信、和底层硬件（WMS）的对接、以及 new ViewRootImpl()（创建包工头），全都是这个全局 Boss 在干！

wm.addView() 到底触发了什么？

当我们调用 wm.addView(decor, l) 时，前面提到的那个**全局 Boss（WindowManagerGlobal）**立刻干了两件事：

  1. new 了一个名叫 ViewRootImpl 的超级包工头。
  2. 调用 root.setView(decor, ...)，把你的皇城（DecorView）正式交给了包工头。

一、幕后黑手 ViewRootImpl 的诞生

别看它的名字里带个 View，它根本不是一个 View！ 它不继承自任何 View 类，它是连接 WindowManager 和 整个 View 树的“绝对纽带”。 可以把它理解为一个专属包工头。

在 Android 里，一个 Activity 对应一棵以 DecorView 为首的视图树；而每一棵视图树，系统都会为它专门配备唯一的一个 ViewRootImpl。

在 WindowManagerGlobal 内部，系统调用了 root.setView(view, ...)。当顺着源码进入到 ViewRootImpl.setView() 这个方法时，会发现这个包工头一上任，立刻干了两件极其硬核的大事。

这就是包工头的“两板斧”：对内搞定渲染节奏，对外搞定系统地皮。

🪓 第一板斧：对内发起渲染请求 (requestLayout)

在 setView 方法的开头，包工头立刻调用了一个极其重要的方法：requestLayout()。

💡 破案时刻： 平时在开发中，如果动态改变了一个 View 的大小（比如把一段很长的文字塞进 TextView），往往需要调用 view.requestLayout()。 这个方法最终都会顺藤摸瓜，一路调用到 ViewRootImpl 的 requestLayout() 这里！

在这个方法里，从大树的根部（DecorView）开始，著名的 Measure（测量）、Layout（摆放）、Draw（绘制）三大战役，正式打响！

二、三大战役 performTraversals()

当 16.6ms 的 VSYNC 信号如期而至，系统底层的 Choreographer（编舞者）准时唤醒了包工头 ViewRootImpl。 此时，包工头极其冷酷地按下了核按钮，执行了 Android UI 渲染史上最著名、最核心的方法：performTraversals()。

扒开这个长达近千行的方法，去掉各种繁杂的状态判断，会发现它的灵魂极其纯粹。包工头带着 DecorView，自顶向下（从大树的根部到每一片叶子），严格按顺序打响了**“三大战役”**：

1. 第一战：测量 (Measure) —— “你有多大？”

包工头调用 performMeasure()，从 DecorView 开始往下发问。

  • 包工头问 LinearLayout：“你需要多大空间？”

  • LinearLayout 转头去问肚子里的 Button 和 TextView：“你们的内容有多宽多高？” 经过一轮极其复杂的计算，整棵树上所有的 View，第一次拥有了确定的宽和高（MeasuredWidth / MeasuredHeight）。

    (💡 这就是为什么在 onCreate 里直接调 view.getWidth() 永远是 0，因为代码跑到那里时，第一战还没打！)

1.1. Measure 第一战的“二进制魔法”

1️⃣极致压缩：什么是 MeasureSpec？

在包工头发起 performMeasure() 时，面临着一个极其严峻的性能考验： 手机屏幕一秒钟刷新 60 次（甚至 120 次），这意味着系统在极短的时间内，要带着成百上千个 View 疯狂地计算大小。如果在测量时频繁地 new 对象来传递数据，瞬间就会触发系统的 GC（垃圾回收），导致极其严重的画面卡顿。

为了抠这点性能，Google 工程师丧心病狂地设计出了 MeasureSpec。

MeasureSpec 翻译过来叫“测量规格”。它并不是一个对象，它是一个极其纯粹的 32 位 int 整型数字！

工程师利用了二进制的位运算（Bitwise Operations），硬生生地把两个完全不同的信息，塞进了一个 int 变量里：

  • 高 2 位（SpecMode）： 代表测量的模式。
  • 低 30 位（SpecSize）： 代表测量的具体大小（像素值）。

当拿着这个 int 数字时，只需要通过系统的按位与（&）和移位（>>）操作，就能瞬间解包出 Mode 和 Size，全程 0 对象分配，性能快到起飞！

2️⃣父与子的谈判：三大 SpecMode

在递归遍历时，父容器（比如 LinearLayout）就是通过 MeasureSpec 向子 View（比如 Button）下达测量命令的。SpecMode 共有三种，它们完美对应了你在 XML 里写的那些属性：

EXACTLY（精确模式）：“强权老爸”

  • 场景：你在 XML 里写了 layout_width="100dp" 或者 match_parent。
  • 父容器潜台词：“儿子，不管你内容有多少，我都已经把你的大小定死了！给你 100 像素，多一点都不行，少一点也不行！”

AT_MOST（最大模式）：“开明老爸”

  • 场景：你在 XML 里写了 layout_width="wrap_content"。
  • 父容器潜台词：“儿子，你先自己算算你需要多大。但我最多只能给你 800 像素（屏幕宽度），只要不超过这个值，你要多大自己定！”

UNSPECIFIED（无限制模式）：“土豪老爸”

  • 场景：极其罕见，通常出现在 ScrollView 或 RecyclerView 的底层测量中。
  • 父容器潜台词：“儿子，老爸我有无限的滚动空间，你想要多大就有多大，哪怕超出屏幕也没关系！”

3️⃣递归回溯：一场自底向上的汇报演出

弄懂了 MeasureSpec 这个“谈判工具”，我们再看测量的真实流程，这是完美的**“递归回溯”**。

测量是一场标准的树形 DFS（深度优先搜索）。 以一个 LinearLayout 里包着一个 TextView 为例，流程是这样的：

  1. 递（向下传达约束）： 包工头给 LinearLayout 下达 MeasureSpec（比如最大 1080 像素）。 LinearLayout 发现自己有儿子，于是它不能马上决定自己的大小。它拿着老爸给的约束，结合 TextView 的 LayoutParams，生成一个新的 MeasureSpec，传给 TextView 的 measure() 方法。
  2. 触底（叶子节点自我测量）： TextView 收到指令。它是叶子节点，没有儿子了。于是它老老实实地调用底层的文本渲染引擎，计算出自己那串文字到底需要占几个像素。算完之后，调用 setMeasuredDimension(width, height)，把结果保存在自己肚子里。
  3. 归（向上回溯汇报）： TextView 测量完毕，出栈。LinearLayout 终于拿到了儿子的真实大小。它把儿子的宽加起来，再加上自己的 padding（内边距），最终算出了自己的总大小，然后也调用 setMeasuredDimension 保存起来。
  4. 一路向上回溯，直到最顶层的 DecorView 算完，整个第一战（Measure）宣告彻底结束！

2. 第二战：布局 (Layout) —— “你站哪儿？”

大小确定了，接下来排座位。包工头调用 performLayout()，再次从上往下发号施令： “LinearLayout，你占据屏幕上半部分！Button，你站在 LinearLayout 内部坐标为 (X:50, Y:100) 的位置！” 打完这一战，所有的 View 都拥有了相对于父容器的四个顶点真实坐标（Left, Top, Right, Bottom）。

3. 第三战：绘画 (Draw) —— “给你画笔，上色！”

万事俱备，只欠上色。包工头调用 performDraw()，拿出之前向系统申请好的 Surface（物理画布），并生成一个 Canvas（画笔）。 包工头大吼一声：“拿到画笔的，依次把自己画在画布上！” 树上的 View 们开始极其有纪律地作画：

  1. 画自己的背景（Background）。
  2. 画自己的内容（比如 TextView 渲染文字，ImageView 画图片）。
  3. 如果自己是容器，就把画笔递给孩子们（dispatchDraw），让孩子们接着画。

4. 上色

当 performDraw() 执行完毕，那块 Surface 画布上已经画满了绚丽的 UI 像素。

但这块画布目前还在 App 的进程内存里。包工头的最后一个动作，是把这块画好的 Surface 缓冲区，一脚踢给系统专门负责合成画面的终极硬件 Boss —— SurfaceFlinger！

SurfaceFlinger 拿到你的画面，再拿到系统顶部状态栏的画面，将它们瞬间合成，直接送往 GPU 和手机屏幕硬件。

三、终极大 Boss SurfaceFlinger

当包工头 ViewRootImpl 极其辛苦地指挥完 Measure、Layout、Draw 三大战役后，UI 像素已经被一笔一画地填满在了那块名为 Surface 的物理画布上。

但此时，面临着一个极其严峻的系统级挑战： App 进程被隔离在沙箱里，它绝对没有权限去直接操作手机的物理硬件屏幕！ 屏幕的绝对控制权，掌握在底层的系统服务进程手中。

那么，App 是怎么把画好的画面交出去的呢？这就轮到 Android 渲染引擎的终极大 Boss —— SurfaceFlinger 登场了！

1. 跨进程的性能狂魔：共享内存 (GraphicBuffer)

很多人会想当然地以为：“既然不在一个进程，那肯定是用 Binder 跨进程把画布传过去呗！” 大错特错！

💡 极客思考：为什么不能用 Binder？ 一个 1080 x 1920 屏幕的一帧画面，如果用 ARGB_8888 格式，大小约为 1080 * 1920 * 4 bytes ≈ 8MB。 而 Android 系统的 Binder 事务缓冲区极限只有 1MB 出头！如果用 Binder 传画面，App 瞬间就会抛出 TransactionTooLargeException 当场崩溃！

为了解决这个物理级难题，Google 引入了极其暴力的 BufferQueue（缓冲队列） 和 匿名共享内存（Ashmem / GraphicBuffer） 机制。

  • App（生产者）： 向 BufferQueue 申请一块内存，在里面画画。画完之后，不需要拷贝庞大的像素数据，只需要通过 Binder 把这块内存的**文件描述符（句柄）**极其轻量地传给系统。
  • SurfaceFlinger（消费者）： 收到句柄后，直接映射这块物理内存。全程零拷贝（Zero-Copy）！ 性能瞬间拉满！

2. 大 Boss 的真正本职工作：图层合成 (Composition)

当 SurfaceFlinger 从队列里拿到 App 画好的那块玻璃（Surface）时，它面临的其实是一堆玻璃：

  • 顶部的系统状态栏（时间、电量）是一块玻璃。
  • 底部的系统导航栏（三大金刚键/小白条）是一块玻璃。
  • App 界面是一块玻璃。
  • 如果你弹出了一个 Toast 或者悬浮窗，那又是一块额外的玻璃。

SurfaceFlinger 的核心工作根本不是“画图”，而是**“合成（Composite）”**！ 它就像一个极其冷酷的千层饼大师，按照 Z 轴（Z-order）的上下顺序，把这些来自不同进程的玻璃板层层叠加上去。如果 App 背景是半透明的，它还会利用 GPU 进行极其复杂的 Alpha 混合运算，算出最终透过去的颜色。

3. 终极一跃：提交给硬件 (HWC)

一切就绪！ 当 VSYNC 信号再次响起，SurfaceFlinger 会把最终合成完毕的一整块“完美画面”，直接提交给硬件层的 HWC（Hardware Composer / 硬件合成器） 和 Display Controller（显示控制器）。

控制器将数字信号转化为极其微弱的电流，以极光般的速度扫过手机屏幕上的几百万个发光二极管（OLED/LCD 像素点）。

至此，在 MainActivity 里敲下的那行极其普通的 setContentView(R.layout.activity_main)，跨越了生命周期、跨越了 View 树的重重解剖、跨越了 Binder 进程边界、跨越了共享内存，最终化作了用户指尖那一抹最绚丽的色彩！

基础的网络知识

网络基础是考察程序员底层内功的试金石。今天系统性梳理一下 HTTP 和 TCP/IP 协议族中那些最重要的知识点

一、TCP/IP网络结构

计算机网络体系结构通常有三种经典的划分方式：OSI 七层模型、TCP/IP 四层模型，以及结合两者优点的五层协议（教学常用）。

在实际工业界试中，我们最常讨论的是 TCP/IP 四层模型。它们之间的对应关系如下表所示：

OSI 七层模型 TCP/IP 四层模型 核心功能与常见协议
应用层、表示层、会话层 应用层 (Application) 为用户应用提供接口 (HTTP, HTTPS, DNS, FTP)
传输层 传输层 (Transport) 提供端到端的通信连接 (TCP, UDP)
网络层 网络层 (Internet) 负责数据包的路由和转发 (IP, ICMP)
数据链路层、物理层 网络接口层 (Network Access) 负责底层硬件的物理传输 (Ethernet, Wi-Fi, ARP)

下面拿 TCP/IP 四层模型做详细说明。理解这四层，是后续排查网络疑难杂症的基础。

1. 应用层 (Application Layer)

用户日常访问的网页、使用的 App 几乎都是直接在应用层上实现的。它只关心为用户提供功能，不关心底层数据如何传输。

  • 常见协议：HTTP、HTTPS、DNS、FTP 等。
  • 开发映射：我们在业务中使用的 Retrofit、OkHttp 发起的网络请求，或者用 WebView 加载网页，本质上都是在构建和解析应用层的数据报文。

2. 传输层 (Transport Layer)

应用层决定了“要传什么”，而传输层则是为应用层提供端到端的网络支持，关心“怎么收发数据”。

  • 常见协议：TCP（保证可靠交付）、UDP（追求极致速度）。
  • 开发映射：如果我们要在 App 里实现 IM 即时通讯或者直播推流，底层用到的 Socket 编程就是直接在这层与操作系统内核打交道。

3. 网络层 (Internet Layer)

数据要从一台手机跨越千山万水到达远端服务器，中间要经过极其复杂的路由器和交换机。网络层负责寻址和路由选择，决定数据包怎么从源设备一步步“跳”到目的设备。

  • 常见协议：IPv4、IPv6、ICMP（平时排查弱网用的 ping 命令就基于此）。

4. 网络接口层 (Network Access Layer)

数据要彻底实现从一台设备通过 Wi-Fi 或蜂窝网络到达另一台设备，需要最底层的链路级别支持，它负责将上层的数字信号转换为物理介质中的电信号或无线电波。

  • 常见协议：ARP 协议（通过 IP 获取网卡 MAC 地址）、以太网协议。

二、在浏览器输入网址后发生了什么？

这是一道可以无限深挖的综合性问题。从宏观的应用层来看，整个过程的第一步是封装请求：

1. URL 解析

当我们在浏览器输入一个 URL 时，浏览器首先要对其进行解析，拆解出目标地址和所需资源。

URL 标准结构 = 访问协议 (http/https) + :// + Web 服务器名称 (域名) + 资源路径

2. DNS 域名解析 (获取目标 IP)

解析完 URL 后，浏览器需要根据域名去寻找目标服务器的真实 IP 地址。寻找 IP 的过程有一套严格的降级查找机制：

  1. 查本地缓存：浏览器自身缓存 -> 操作系统缓存 -> 本地 hosts 文件。
  2. 查 DNS 服务器：如果本地都没有记录，就会向本地域名服务器发送 DNS 请求（通常会经过根 DNS 服务器、顶级 DNS 服务器、权威 DNS 服务器的查询过程）。

3. 封装 HTTP 请求消息

拿到了目的 IP，浏览器就可以正式封装要在应用层发送的 HTTP 请求报文了。

HTTP 消息的结构非常严谨，通常由四个部分组成：

HTTP 消息 = 请求行/响应行 + 请求头 (Headers) + 空行 (CRLF) + 请求体 (Body)

三、各个协议部分细节

1. HTTP

1.1 http的请求方法

HTTP 请求行的开头就是请求方法。官方定义的方法类型有多种，这里以面试中最常被对比的 GET 和 POST 为例说明：

  • GET：一般用于获取资源。它对服务器资源不会发生更改，具有幂等性。例如我们请求接口查看“今天是星期几”， 无论用户查看多少次，都不影响结果和服务器状态。GET 的请求参数通常直接暴露拼接在 URL 中（例如 /?zone=zh）。
  • POST：一般用于操作/提交资源。它通常会改变服务器状态，不具有幂等性。例如电商付款，用户如果 （因网络卡顿等原因）重复发起了 N 次 POST 扣款请求，如果没有做特殊处理，账户余额就会被扣款 N 次， 商家也会新增 N 个订单。 POST 的参数通常封装在请求体 (Body) 中。

1.2 HTTP 的状态码

HTTP 响应报文的开头就是状态码。虽然在实际业务开发中，我们经常会在响应体（Body）里与后端自定义业务状态码，但在底层的 HTTP 协议层面，官方推荐的标准分类是：

  • 1xx (信息提示)：请求已被接收，还需要后续处理。
  • 2xx (成功)：服务器已成功接收并处理了请求。
  • 3xx (重定向)：需要客户端采取进一步的操作才能完成请求。
  • 4xx (客户端错误)：客户端的请求有误，服务器无法处理。
  • 5xx (服务端错误)：服务器在处理请求的过程中发生了内部错误。

1.3 HTTP 版本的演进

  • HTTP/1.0：短连接。每次发送请求都需要重新建立 TCP 连接（即经历完整的“三次握手”和“四次挥手”），网络开销极大。
  • HTTP/1.1：长连接（Keep-Alive）。默认开启长连接，允许多个 HTTP 请求复用同一个 TCP 连接，省去了频繁连接的开销。
    • 致命痛点：存在**“队头阻塞 (Head-of-line blocking)”**。在一个 TCP 连接中，如果前面的请求响应太慢，后面的请求只能排队干等。
  • HTTP/2.0：性能的全面跨越。
    • 多路复用（Multiplexing）：彻底解决了 HTTP 层的队头阻塞。多个请求可以同时在一个 TCP 连接上并发、交错地发送和接收，谁先处理完谁就先回来，极大提升了加载速度。
    • 头部压缩（HPACK）：HTTP/1.1 每次都会发送大量重复的 Header（如 User-Agent、Cookie），HTTP/2 引入了字典机制，把头部压缩得非常小。
    • 二进制分帧：放弃了纯文本传输，改用更高效、更易于机器解析的二进制格式。
  • HTTP/3.0 (面试加分项)：底层协议的革命。因为 HTTP/2 依然基于 TCP，TCP 本身的丢包重传机制会导致 TCP 层面的队头阻塞。HTTP/3 直接抛弃了 TCP，底层改用基于 UDP 的 QUIC 协议。它不仅连接速度极快，而且在移动端经常切换网络（比如从 Wi-Fi 切到 4G）时不会断线重连，这对于客户端的弱网优化意义重大。

2. HTTPS：给“裸奔”的 HTTP 穿上铠甲

HTTP 默认是明文传输的，无论是把参数拼在 URL 里还是放在 Body 里，在抓包工具和中间路由器眼里都是在“裸奔”。为了解决安全问题，HTTPS 应运而生。

HTTPS = HTTP + TLS/SSL（安全层）

HTTPS 并不是一种全新的协议，而是把 HTTP 原本直接交给 TCP 的明文数据，先交给 TLS 层进行加密，再由 TCP 传输。它通过三大核心机制保障安全：

  1. 混合加密（机密性）：采用“非对称加密”来安全地传输密钥，再用“对称加密”来高效地加密实际的业务数据。
  2. 摘要算法（完整性）：类似数据的“指纹”，防止数据在传输过程中被黑客篡改。
  3. 数字证书（身份认证）：通过 CA 机构颁发的证书，加上“链式校验”，防止中间人冒充目标服务器。

2.1 TLS 握手到底是怎么协商出密钥的？

(以经典的 TLS 1.2 / RSA 密钥交换为例，TCP 三次握手完成后，开始 TLS 握手)

  1. 客户端打招呼 (Client Hello)：客户端向服务端发送支持的加密套件列表，并生成一个 客户端随机数 (Random 1)。
  2. 服务端回应 (Server Hello + Certificate)：服务端确认加密算法，生成一个 服务端随机数 (Random 2)，并把自己的**数字证书（内含服务器公钥）**发送给客户端。
  3. 客户端验证与生成密钥：
    • 客户端使用本地内置的 CA 机构公钥，验证服务器证书的真实性。
    • 验证通过后，客户端生成第三个随机数，称为 Pre-Master Secret (预主密钥)。
    • 客户端用服务器的公钥将这个预主密钥加密，发送给服务端。
  4. 服务端解密：服务端收到后，用自己的私钥解密，得到 Pre-Master Secret。
  5. 生成最终对称密钥：此时，双方手里都有了同样的三个要素：Random 1 + Random 2 + Pre-Master Secret。双方使用同样的算法，用这三个要素计算出最终的**“会话密钥”**。

至此，握手完成。接下来双方的 HTTP 通信，都只用这个“会话密钥”进行极其高效的对称加解密，黑客即使抓包也只能看到一堆乱码。

2.2 既然 HTTPS 这么安全，为什么 Charles/Fiddler 还能抓到密码？

很多人了解了HTTPS后会有疑问：“既然 HTTPS 加密了，为什么我们在开发时用抓包工具依然能看到明文数据？抓包的原理是什么？”

答案是：抓包本质上是一次“合法的中间人攻击 (MITM)”，核心在于你主动信任了代理工具的 CA 证书。

正常的直连流程是：客户端 <====> 服务器。
而打开抓包工具（如 Charles）后，流程变成了：客户端 <====> Charles <====> 服务器。

具体的“欺骗”过程如下：

  1. 拦截请求：客户端向真实服务器发起 HTTPS 连接，Charles 在中间拦截了这个请求。
  2. 伪造证书：Charles 伪装成服务器，给自己动态签发了一张伪造的数字证书，发给客户端。
  3. 关键的一步 (CA 的作用)：正常情况下，客户端（手机）发现这张证书不是正规机构（如 DigiCert）签发的，会直接报“证书不可信”并断开连接。但是，我们在配置抓包环境时，都会手动把 Charles 的根证书 (Root CA) 下载并安装到手机的“受信任的凭据”列表中。
  4. 握手成功：客户端校验时，发现这本伪造证书的根证书竟然在系统信任列表里！于是乖乖地用 Charles 的公钥加密了预主密钥（Pre-Master Secret）。
  5. 双向解密：Charles 用自己的私钥解开拿到密钥，这就和客户端建立了第一段加密连接。随后，Charles 再以客户端的身份，去和真正的服务器完成第二段标准的 HTTPS 握手。

最终，所有数据都在 Charles 这里经历了一次**“解密 -> 记录明文 -> 重新加密”**的过程。

💡Android 7.0 的网络安全限制
面试官如果深挖，一定要抛出这个知识点：从 Android 7.0 (API 24) 开始，系统默认不再信任用户手动安装的 CA 证书（只信任系统内置的）。
这就是为什么在新版 Android 手机上抓包，经常会遇到网络请求直接 Crash 或者抓不到 HTTPS 的原因。解决办法是在 Android 工程的 res/xml/network_security_config.xml 中，显式配置 <trust-anchors>，允许在 debug 模式下信任用户证书。

2. TCP 核心解析

应用层（如 HTTP）往往不管怎么收发数据，网络层（如 IP）往往不管是否丢包、是否乱序，而 TCP 就是来兜底保障数据可靠传输的（提供可靠性和流量控制）。

TCP = 面向连接 + 可靠 + 字节流

  • 面向连接：一对一通信（相对的，UDP 可以一对多）。
  • 可靠：无论网络链路发生什么变化，保证数据包可以被完整传送到目的地。
  • 字节流：通过 TCP 协议传输时，消息可能会被操作系统分组成多个 TCP 报文，字节就是最小单位，它没有固定的业务边界。

2.1 TCP 与 UDP 的报文结构对比

TCP 报文核心字段：

  • 序列号 (Sequence Number)：建立连接时生成的随机数字作为初始值。发送端每发送一次数据，就累加一次该数据字节数的大小。（核心作用：解决网络乱序问题）
  • 确认应答号 (Acknowledgment Number)：期望下一次收到的序列号。（核心作用：解决丢包问题）

浅提 UDP：

  • UDP 报文 = (源端口，目标端口) + (包长度，校验和) + 数据。
  • 它是没有复杂控制机制的、面向 IP 的无连接通信服务。

2.2 TCP vs UDP

对比维度 TCP UDP
连接性 面向连接 无连接
通信对象 严格的一对一 一对一、一对多、多对多
可靠性 可靠交付 最大努力交付（可用 QUIC 在应用层补救）

传输形态 面向字节流（无边界，保证有序） 面向数据报（有边界，可能乱序）
分片策略 数据大于 MSS 会在传输层分片 数据大于 MTU 会在 IP 层 分片
首部开销 较长 (通常 20 字节以上) 极小 (仅 8 字节)

2.3 图解 TCP 三次握手

  1. 第一次握手：客户端发送 SYN 报文（携带随机生成的 client_seq），进入 SYN_SENT 状态。
  2. 第二次握手：服务端接收后，发送 SYN + ACK 报文（确认号为 client_seq + 1，并携带自己的 server_seq），进入 SYN_RCVD 状态。
  3. 第三次握手：客户端发送 ACK 报文（确认号为 server_seq + 1），进入 ESTABLISHED (稳定连接) 状态。此时的 ACK 报文是可以携带业务数据的。
    (服务端收到 ACK 后，也进入 ESTABLISHED 状态)

灵魂拷问 1：为什么是三次握手？两次不行吗？

  • 防止历史连接（最核心原因）：防止旧的迟到连接请求突然到达服务端，导致服务端错误建立连接。
  • 同步双方序列号：确保双方都有机会发送和确认彼此的初始序列号。
  • 避免资源浪费：防止服务端因失效请求而一直白白分配内存等资源。

灵魂拷问 2：为什么初始序列号要随机？

如果每次都使用相同的初始序列号，那么网络中滞留的历史报文极有可能会被下一个相同四元组的新连接错误接收。(注：序列号的生成基于时钟，每 4 微秒 +1，极难重复)。

2.4 图解 TCP 四次挥手

当客户端和服务端的数据传输完毕后，就需要断开连接。断开连接的过程需要经过四次交互，也就是常说的“四次挥手”。

  1. 第一次挥手：客户端发送 FIN 报文（表示自己没有数据要发了），进入 FIN_WAIT_1 状态。
  2. 第二次挥手：服务端收到 FIN 报文后，回复 ACK 报文，进入 CLOSE_WAIT 状态。
    • （此时客户端收到 ACK 后，进入 FIN_WAIT_2 状态。此时 TCP 连接处于“半关闭”状态，即客户端不发了，但服务端如果还有数据，依然可以继续发给客户端）。
  3. 第三次挥手：服务端的数据也发送完毕了，于是发送 FIN 报文给客户端，进入 LAST_ACK 状态。
  4. 第四次挥手：客户端收到服务端的 FIN 报文，回复 ACK 报文，进入 TIME_WAIT 状态。
    • (服务端收到这个 ACK 后，立刻进入 CLOSED 状态，彻底关闭连接)。
  5. 最终关闭：客户端在 TIME_WAIT 状态等待 2MSL（报文最大生存时间的 2 倍）后，如果没有收到服务端的重传，才会进入真正的 CLOSED 状态。

灵魂拷问 1：为什么建立连接是三次，挥手却需要四次？

因为 TCP 是全双工通信，存在**“半关闭”**状态。单向发送的 FIN 只代表该方的数据发送完毕，不代表对方也没有数据要发送。因此，服务端的 ACK（确认收到断开请求）和 FIN（服务端自己也准备好断开了）通常是分两步发送的。

灵魂拷问 2：为什么主动关闭的一方需要 TIME_WAIT 状态，且要等待 2MSL？

这是面试官极其喜欢深挖的考点：

  • 防止历史数据干扰：保证网络中该连接产生的哪怕是最迟到的数据包，也都在这 2MSL 内自然消亡。防止它们被后续使用相同“四元组”的新连接错误接收。
  • 保证被动关闭的一方能可靠地断开：如果客户端最后发送的那个 ACK 报文在网络中丢失了，服务端会因为没收到确认而重发 FIN。TIME_WAIT 状态确保客户端有足够的时间去接收这个重传的 FIN 并再次回复 ACK。

3. WebSocket：真正的全双工自由

有不少人存在一个直击灵魂的疑惑：TCP 本身是全双工的，但基于 TCP 的 HTTP 却是一问一答的（而且只能由客户端主动发问，服务器被动应答），这难道不是半双工吗？
❓❓❓ ❓❓❓ ❓❓❓

其实，前辈们早就意识到了 HTTP 这个“不主动”的缺陷，所以在 WebSocket 诞生之前，业界妥协出了“短轮询”和“长轮询”的方案。比如很多网站的“扫码登录”页面，底层用的就是长轮询。

  • 短轮询（Short Polling）：客户端像个复读机一样不断发送小请求（“我登录成功了吗？”“现在呢？”“成功了吗？”…），极其消耗服务器和网络资源。
  • 长轮询（Long Polling）：HTTP 请求发送后，服务器如果没数据，就先憋着不回响应，把这个连接挂起（设置较长的超时时间）。这样就留给服务器足够的时间去处理业务，一旦用户在手机上点了确认，服务器立刻顺着这条挂起的连接返回“登录成功”。

虽然有这两种方法，但是对于需要高频、双向交互的场景，轮询就显得极其鸡肋了。比如你逛着某宝，不小心点到了右下角弹出的绚丽广告，打开了一个“一刀999”的页游。你甚至什么都不用操作，屏幕里的角色自己就开始疯狂打怪、掉装备、全服广播。这种服务器需要高频向客户端主动推送数据的场景，WebSocket 它来了！

3.1 WebSocket 的握手与数据帧

WebSocket 是应用层的协议，它的唯一目的就是实现真正的全双工通信。

  • 借鸡生蛋的握手：WebSocket 的建立是“踩在 HTTP 的肩膀上”完成的。TCP 三次握手完成后，客户端会发送一个特殊的 HTTP 请求（Header 里带有 Upgrade: websocket）。服务器如果支持，就会返回一个经典的 101 Switching Protocols (协议转换) 状态码。这一来一回的 HTTP 握手结束后，这条连接就正式升级为 WebSocket，从此以后和 HTTP 再无半点瓜葛。
  • 数据帧 (Frame)：WebSocket 传输的数据不叫报文，叫“帧”。它的数据格式设计得非常精巧，可以粗略理解为 数据头 + 消息体。因为消息头里明确包含了消息体的长度，所以接收方能够精确地切分数据，这也就极其优雅地解决了底层 TCP 的粘包问题。

4. RPC：把“网络调用”伪装成“本地调用”

总有人喜欢问一个极其宽泛且带有误导性的问题：“既然有了 HTTP，为什么还要有 RPC？”

很多初学者不知道怎么回答，是因为没看透本质。

核心真相：RPC 根本不是某一个具体的网络协议，而是一种“架构设计理念”。

  • HTTP 是一种具体的通信规范（规定了请求头、状态码、报文长什么样）。
  • RPC (Remote Procedure Call，远程过程调用) 是一种行为。它的终极目的是**“让调用远程服务，就像调用本地方法一样简单”**。像 gRPC、Thrift、Dubbo 这些具体的 RPC 框架，就是为了帮你把底层的网络打包、寻址、序列化等脏活累活全封装起来。

4.1 灵魂反击：如何回答“有了 HTTP 为什么还要 RPC”？

如果在遇到这个问题，你是否和我一样不知从何讲起：

  1. 两者根本不在一个维度，甚至可以融合
    • RPC 的底层传输层不仅可以基于 TCP 直接写，它完全也可以基于 HTTP 协议！ 比如目前极其主流的 Google gRPC，它的底层就是雷打不动的 HTTP/2 协议。
  2. 应用场景不同（内外有别）
    • HTTP 主外（B2C）：像浏览器、手机 App 访问后端服务器，面对的是全世界各种各样的客户端。它需要跨语言、跨平台，主打一个“通用、标准、无脑兼容”。所以 HTTP 往往使用 JSON/XML 这种可读性强但体积大的文本格式。
    • RPC 主内（B2B）：通常用于公司内部的微服务与微服务之间的通信（比如阿里的订单服务调用库存服务）。既然都是“自家程序员写的代码”，追求的就是极致的效率。所以 RPC 框架往往采用极致压缩的二进制流（如 Protobuf），抛弃了臃肿的 HTTP Header，体积更小，速度飞快。
  3. 开发体验的极致追求
    • 写 HTTP 请求，哪怕用 Retrofit，你也得关心 URL 地址、请求方法（GET/POST）、HTTP 状态码解析。
    • 用 RPC 框架，你只需要在代码里写 userService.getUserInfo(uid)，至于这行代码是怎么变成二进制、怎么跨越千山万水飞到另一台机房的服务器上并带回结果的，你完全不需要关心。这就是 RPC 的魅力。

总结：在真实的 Android 开发中，虽然有 Retrofit 和 OkHttp 帮我们封装好了底层的网络请求，但理解这些协议栈的细节，是我们进行弱网优化、排查连接超时问题的根本所在。

Android四大组件之Activity详解

一、 什么是 Activity？

Activity 是 Android四大组件之一，是一种可以包含用户界面的组件，主要用于和用户进行交互。

1. Activity 的基本用法

要让一个 Activity 真正跑起来，必须经过以下几个关键步骤：

  1. 创建： 继承 AppCompatActivity 并重写 onCreate() 方法。
  2. 绑定视图： 在 onCreate() 中调用 setContentView(R.layout.xxx) 加载布局。
  3. 注册（极其重要）： 所有的 Activity 必须在 AndroidManifest.xml 中进行注册，否则启动直接崩溃。

2. Activity 的生命周期（核心重点）

它描述了一个界面从诞生到销毁的完整过程。

主要包含 7 个核心回调方法：

  • onCreate()：Activity 正在被创建（初始化数据、加载布局）。
  • onStart()：Activity 正在启动（即将可见，但还不能交互）。
  • onResume()：Activity 已经可见，并且获取了焦点（用户可以开始疯狂点击了）。
  • onPause()：Activity 正在暂停（通常是因为弹出了一个对话框，或者正准备去别的界面。此时可见，但失去焦点）。
  • onStop()：Activity 即将停止（完全不可见，被压在后台了）。
  • onDestroy()：Activity 即将销毁（回收资源，寿终正寝）。
  • onRestart()：Activity 重新启动（从后台重新回到前台时触发）。

3. Activity 的启动模式 (LaunchMode)

在复杂的业务场景中（比如点通知栏、或者连续打开同一个界面），我们需要配置不同的启动模式来管理 Activity 返回栈（Task Stack）。

四种主流启动模式：

  1. Standard (标准模式)：默认模式，不管栈里有没有，每次都无脑 new 一个新实例。
  2. SingleTop (栈顶复用)：如果它已经在栈的最顶端，就直接用，不新建；如果不在顶端，还是会新建。
  3. SingleTask (栈内复用)：极其霸道。只要栈里有它，就把它上面所有的 Activity 全部踢出栈，让它重见天日（常用于 App 的 MainActivity）。
  4. SingleInstance (单实例)：最孤独的模式。系统会专门为它开辟一个新的返回栈，并且这个栈里只能有它一个人（常用于系统的通话界面、闹钟界面）。

老祖宗 ComponentActivity 搭建好了两大核心基础设施：

  1. ContextAwareHelper：负责监听 Context 的注入时机。
  2. SavedStateRegistry：负责管理系统杀后台时的状态保存与恢复。

    • 核心动作： 注册监听 ➡️ 抢在 onCreate 前安装 ViewFactory 拦截器 ➡️ 打包/恢复主题与夜间模式状态 ➡️ 偷梁换柱替换原生 View。

  • 🧩 处理 Fragment 碎片恢复 (FragmentActivity)

    • 大管家： FragmentManager (底层被包装在 FragmentController 中)

四、 ComponentActivity

🧩 ComponentActivity：全能的“斜杠”基座

在源码中，ComponentActivity 的定义长得惊人，因为它实现了超过 10 个接口。这意味着它不仅是一个 Activity，更是多种能力的提供者（Owner）：

  • 接口化设计：它不直接实现逻辑，而是通过实现接口（如 LifecycleOwner），将能力委派给内部的 Helper 类。
  • 默认开启 (open)：作为架构基石，它被标记为 open class，允许 FragmentActivity 等后续子类不断对其进行功能扩容。

深度思考：为什么不把所有功能都写在 Activity 原生类里？
答案：解耦。通过接口化，Google 可以让 Fragment 甚至自定义 View 也实现这些接口（如 LifecycleOwner），从而拥有一套通用的架构规范。

子类ComponentActivity(FragementACtivity的直接父亲)是androidx.activity包，和androidx.core.app.ComponentActivity()只是类名同，这种设计哲学在Android中非常常见。

  1. androidx.core.app.ComponentActivity非常的轻量，源码（包括注释）就一百多行。

它唯一存在的目的就是解决最基本的系统兼容性问题（比如在老版本手机上模拟新的 KeyEvent 传递），几乎不涉及任何复杂的 Jetpack 架构（没有 ViewModel，没有 SavedState 等）。

  1. androidx.activity.ComponentActivity开启了“疯狂叠 Buff”模式

它实现了 LifecycleOwner、ViewModelStoreOwner 等那一长串接口。这才是我们平时真正使用的、具备完整 Jetpack 能力的 Activity 基座

  • 核心架构战队（Jetpack 的地基）

    • LifecycleOwner：生命周期持有者。让 Activity 拥有了“感知生死”的能力。你可以直接观察它的生命周期，而不需要在 onStart/onStop 里写一堆逻辑。

    • ViewModelStoreOwner：ViewModel 仓库。让 Activity 有能力在配置更改（如旋转屏幕）时，依然能死死抓住 ViewModel 实例不放。

    • HasDefaultViewModelProviderFactory：默认工厂。它提供了一个默认的“厨师”，当你调用 by viewModels() 时，就是这个工厂帮你实例化 ViewModel 的。

    • SavedStateRegistryOwner：状态恢复官。这就是咱们之前聊的那个“寄存处”，负责在 App 被杀掉后恢复那点“临终记忆”。

  • 事件分发战队（告别逻辑耦合）

    • OnBackPressedDispatcherOwner：返回键分发者。你可以通过 onBackPressedDispatcher 给不同的 Fragment 单独设置返回逻辑，不用再在 Activity 里写一堆 if/else。
    • MenuHost：菜单宿主。现在的菜单（Toolbar 上的三个点）不再必须写在 Activity 里，通过它，Fragment 也可以自主管理自己的菜单。
    • NavigationEventDispatcherOwner：导航事件分发。专门处理 Activity 级别的导航事件传递。

  • 系统回调战队（感知外部环境）

    • OnConfigurationChangedProvider：当屏幕旋转、字体变大、语言切换时，通知感兴趣的人。
    • OnTrimMemoryProvider：内存整理通知。系统内存不足时，通知各个组件赶紧清理缓存，别被系统杀了。
    • OnMultiWindowModeChangedProvider / OnPictureInPictureModeChangedProvider：窗口模式感知。处理分屏和画中画模式的切换。
    • OnNewIntentProvider：当 Activity 的启动模式为 singleTask 重新进入时，通知新意图。

  • 权限与结果战队（再见 startActivityForResult）

    • ActivityResultRegistryOwner & ActivityResultCaller：结果注册中心。 这是 ActivityResultLauncher 的底层支撑。让你用“注册回调”的方式去请求权限、拍照、选图片，彻底废弃了那个乱糟糟的 requestCode 匹配。

  • 其他特殊战队

    • ContextAware：上下文感知。正如我们之前聊的，确保在 onCreate 之前，大家能监听到 Context 已经准备好了。
    • OnUserLeaveHintProvider：离场提示。当用户按 Home 键导致 Activity 进入后台时的回调。
    • FullyDrawnReporterOwner：绘制报告者。用于性能优化，精确上报 Activity 什么时候才算“完全绘制完成”，帮助测量冷启动耗时。

这部分内容太丰富了，其中包含了非常多巧妙的实现，后面会分别开新文章做详细分析。

五、Activity

无论是早期的Android还是现代化的Activity，所有Activity的祖先必定是android.app.Activity。虽然Google设计的ComponentActivity已经让Activity不那么臃肿，单其实android.app.Activity内容是非常多的，这个类的源码有近万行，是一个非常大的类（Android称霸世界也不是只靠时运，也是不乏实力的）。

（如果是要逐字阅读这一万行代码，我估计我会疯掉。看了很多技术博客和一些技术书籍，形成了浅显、片面且带有极强主观看法的看法。）

Activity主要有四个核心点，核心生命周期（状态机）、窗体管理（View 的宿主）、交互分发（接线员）、组件间通讯（信使）。

  • 核心生命周期（状态机）：performCreate, performStart, performResume。
    • 系统（ActivityThread）是通过这些 performXXX 方法来驱动 onCreate, onStart 的。这就是 Activity 的“心跳”
  • 窗体管理（View 的宿主）: mWindow, setContentView, attach
    • Activity 本身不显示东西，它内部持有一个 PhoneWindow（大管家）。当你调用 setContentView，其实是把布局塞给了这个 Window。
  • 交互分发（接线员）：KeyEvent.Callback, Window.Callback
    • 所有的按键点击、触摸屏幕，都是先传给 Activity，再由它分发给 Window 或 View。
  • 组件间通讯（信使）： startActivityForResult, onActivityResult。
    • 它是怎么和别的 Activity 打交道的

二、安卓项目结构

如果使用Android Studio新建一个普通的安卓项目，初始的内容主要包括：

1. 编译器相关文件

  • .gradle, .idea, .iml：这些是IDE（编译器）自动生成的配置文件，通常在使用Git进行版本控制时会被忽略。

2. 核心代码目录

  • app：项目的代码、资源等内容全都在 app 目录下，这是编写应用程序主要的工作目录。核心包含：
    • src/main/java：存放所有的Java/Kotlin源代码。
    • src/main/res：存放各种资源文件，如UI布局（layout）、图片（drawable/mipmap）和常量配置（values）。
    • src/main/AndroidManifest.xml：应用的全局配置清单，包含权限声明（如网络、存储）和四大组件注册。

3. 构建与配置系统

  • gradle文件夹：包含Gradle Wrapper，用于统一团队的构建环境版本。
  • build.gradle：极其重要，分为项目级和模块级。用于配置SDK版本、包名（applicationId）以及引入第三方依赖库。
  • gradle.properties：全局的Gradle配置文件，可设置编译内存等优化参数。
  • gradlew / gradlew.bat：命令行执行脚本（分别对应Mac/Linux和Windows）。
  • settings.gradle：指定项目包含了哪些模块（Module），多模块开发时必用。
  • local.properties：指定本地Android SDK的绝对路径。由于每个人电脑路径不同，该文件绝对不能提交到Git仓库中。

三、app 模块核心目录

app 是整个安卓项目中最核心的模块（Module），我们日常 99% 的代码开发工作都在这里完成。展开它，你会看到以下结构：

  • build：构建输出目录。每次编译/运行项目时，系统自动生成的 .class 编译文件、最终打包出的 APK 安装包等都存放在这里。无需手动修改，通常在 Git 中会被忽略。
  • libs：本地依赖库目录。如果有一些第三方的 .jar 或 .aar 包没法通过 Gradle 在线依赖（比如某些 SDK），就可以直接丢到这个文件夹里引入。
  • androidTest：Android 自动化测试目录。用来编写必须运行在真机或模拟器上的 UI 测试代码（比如模拟用户点击按钮、滑动屏幕）。
  • java：源代码主目录。注意，虽然它的名字叫 java，但实际上你编写的 Java 和 Kotlin 源代码文件全都在这里面。
  • res：资源大本营 (Resources)。包含了所有非代码的静态资源，比如：
    • layout：XML 界面布局文件。
    • drawable：图片资源、XML 绘制的形状资源。
    • mipmap：各个分辨率的 App 桌面图标。
    • values：字符串 (strings.xml)、颜色 (colors.xml)、样式主题 (themes.xml) 等常量配置。
  • AndroidManifest.xml：应用全局配置清单。极其重要！App 的入口页面、四大组件（Activity、Service 等）的注册、以及各种系统权限（如联网、相机、位置）的申请，统统都要在这里向系统“报备”。
  • test：本地单元测试目录。用来编写直接运行在电脑本地 JVM 上的测试代码。它不依赖 Android 环境，主要用来测试纯逻辑的算法代码，运行速度极快。
  • .gitignore：Git 忽略清单。专门指定当前 app 模块下有哪些文件和目录不需要提交到代码仓库（比如 build 文件夹）。
  • app.iml：IDE 模块配置文件。由 IntelliJ IDEA / Android Studio 自动生成和维护，记录了模块的路径和依赖关系，开发者通常不需要关心它。
  • build.gradle：模块级构建脚本。这是日常开发打交道最多的文件之一。里面配置了当前 App 的包名 (applicationId)、版本号 (versionCode / versionName)、编译兼容 SDK 版本，以及极其重要的第三方开源库依赖 （dependencies）。
  • proguard-rules.pro：代码混淆与压缩规则文件。当我们准备打包 Release 正式版发布到应用市场时，为了减小包体积并防止黑客反编译我们的代码，会开启混淆。这个文件就是用来告诉系统：“哪些核心类和方法必须保留原名，绝对不能被混淆”。
